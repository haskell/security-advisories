# Hackage CSRF vulnerability

- Vulnerable File: `src/Distribution/Server/Features/Votes.hs` (example)
- Impact: can forge requests through XSS

hackage-server lacked Cross-Site Request Forgery (CSRF) protection
across its endpoints. Scripts on foreign sites could trigger requests to
hackage server, possibly abusing latent credentials to upload packages
or perform other administrative actions. Some unauthenticated actions
could also be abused (e.g. creating new user accounts).

To fix the issue, a new CSRF middleware checks all requests. Requests
using HTTP methods other than `GET`, `HEAD` and `OPTIONS` are subject to
a check of the [`Sec-Fetch-Site`
header](https://developer.mozilla.org/en-US/docs/Web/HTTP/Reference/Headers/Sec-Fetch-Site),
which is [widely supported by modern
browsers](https://caniuse.com/?search=sec-fetch-site). Cross-site
requests are `403 Forbidden`. Certain approved and expected non-browser
user agents (e.g. `cabal-install/*`) are exempted from the check, as are
requests using token authentication (`Authorization: X-ApiKey ...`).

The fix has been
[committed](https://github.com/haskell/hackage-server/commit/2de3ae45082f8f3f29a41f6aff620d09d0e74058)
and deployed on `hackage.haskell.org`.
